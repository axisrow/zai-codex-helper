<objective>
Deliver the matched `install-service` / `uninstall-service` pair that manages the Moon
Bridge LaunchAgent, using the modern `launchctl bootstrap` / `bootout` API (NOT the
deprecated `load`/`unload`). Both commands share ONE plist Label constant (imported from
`PlistBackend.LABEL`) so uninstall never orphans a registered agent. Post-install
verification proves the agent is actually loaded (`launchctl print`) and listening (a TCP
port probe), not just that `bootstrap` exited 0. This is the step `setup` (Phase 12)
OFFERS — Phase 13 owns the actual launchctl calls + plist write.

Purpose: Complete the service-lifecycle half of the Core Value loop. After `setup` builds
the binary and `use zai` wires the provider, the user runs `install-service` once to make
Moon Bridge a persistent, login-started, always-alive LaunchAgent — and `uninstall-service`
to cleanly remove it. SERV-01/02/03/04 (D-83..D-88) are all load-bearing here.
Output: `src/zai_codex_helper/services/lifecycle.py` (install_service, uninstall_service,
verify_service_loaded, LAUNCHAGENT_LABEL), real handlers in `cli/parser.py`, and a
mocked-runner unit-test suite in `tests/test_service_lifecycle.py`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user CLI → handler | Untrusted argv crosses here (argparse validates shape; no untrusted data reaches launchctl argv — every token is a fixed literal or derived from `os.getuid()` / `Paths`) |
| helper → launchctl (subprocess) | The helper invokes `launchctl` as a subprocess; argv is constructed from constants + `os.getuid()` + resolved `Paths` (no shell, no user interpolation) |
| helper → filesystem (`~/Library/LaunchAgents/`) | Mutating writes/unlinks to the per-user LaunchAgents dir |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

NO `shell=True`, NO user-supplied string interpolation. The `<LABEL>` is the imported `PlistBackend.LABEL` constant (D-85) — cannot be drifted by a caller. |
