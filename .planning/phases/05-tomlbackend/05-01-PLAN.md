<objective>
Deliver `TomlBackend` — the first concrete `ConfigBackend` (Phase 4 ABC) and the load-bearing
piece of the entire project. `~/.codex/config.toml` can be parsed with tomlkit, mutated through
a replace-not-append upsert, and written back losslessly: comments, blank lines, key order, and
Codex `[project_*]` trust blocks survive a no-op load → dump round-trip (ROADMAP SC-1). A nested
`[model_providers.*]` upsert replaces an existing sub-table rather than appending a duplicate
(ROADMAP SC-2). This is where `Paths` (Phase 2) + `atomic_write` (Phase 3) + `ConfigBackend`
(Phase 4) + `tomlkit` (Phase 1 dep, first runtime import) all meet for the `.toml` file type.

Purpose: CONF-02 ("Патч config.toml через tomlkit — сохраняет комментарии, порядок ключей и Codex
project trust blocks на round-trip"). If this regresses, every later `use zai` (Phase 7) corrupts
the user's Codex config — hence the round-trip test is the single highest-signal test in the project.

Output: `TomlBackend` class + `upsert_block` helper + a thorough `tests/test_toml_backend.py`
seeding every comment style + a trust block + a nested provider table.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → TomlBackend.read | Untrusted on-disk `config.toml` is read and parsed. A malformed or adversarial TOML could raise parse errors or carry surprising constructs. |
| TomlBackend.write_canonical → filesystem (`~/.codex/config.toml`) | The user's real Codex config is overwritten. A bug here corrupts the user's Codex setup — the highest-impact trust boundary in the project. |
