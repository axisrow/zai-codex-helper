    **Module-level helper `deep_merge(base: dict, override: dict) -> dict`** (D-60 — the recursive dict merge):
    - Returns a NEW dict (do not mutate `base` or `override`).
    - Iterate `override.items()`: for each key/value, if the key is NOT in `base` OR the `base` value is not a dict OR the override value is not a dict, the override wins (overwrite the leaf). If BOTH the `base` value and the override value are dicts, recurse (`deep_merge(base[key], override[key])`). This is a recursive deep-merge: nested dicts are merged key-by-key down to the leaves; non-dict values (including lists, strings, numbers, bools, None) are replaced wholesale by the override (lists are NOT element-merged — they are overwritten; document this, as `models_cache.json` entries are dict-shaped, not list-shaped).
    - Raise `TypeError` if either argument is not a `dict` (defensive — guards against a caller passing a list/string where a dict is expected; the message names the offending argument).

    **Class `JsonBackend(ConfigBackend)`:**
    - `__init__(self, paths: Paths) -> None`: `super().__init__(paths, "models_cache")`.
    - `read(self) -> dict`: if the file exists, `json.loads(self._path.read_text(encoding="utf-8"))` and return the result; the caller expects a `dict` (the file's top level is a JSON object). If the file does NOT exist, return `{}` (an empty cache is the no-entry baseline — cleaner than raising for a fresh user who has never populated `models_cache.json`; the merge path handles empty-as-empty). If the file exists but parses to something that is NOT a `dict` (e.g. a top-level list), raise `ValueError` with a clear message (the merge contract is object-level; a non-object top level is a corrupted or unexpected file — fail loudly, do not silently coerce).
    - `exists(self) -> bool`: `self._path.exists()`.
    - `write_canonical(self, content: dict, mode: int | None = None) -> None`:
      * `content` is the dict to merge IN (the override — e.g. Phase 15's `glm-5.2` entry). Raise `TypeError` if `content` is not a `dict` (defensive — the contract is object-level merge).
      * Read the current state via `read()` (handles file-absent as `{}`).
      * Compute `merged = deep_merge(current, content)`.
      * Serialize via `json.dumps(merged, indent=2)` (pretty-printed, 2-space indent, deterministic key order from Python's dict — `sort_keys` may be left False since deep_merge preserves insertion order of `base` then appends new keys; do NOT set `sort_keys=True` unless deterministic ordering across runs is desired — leaving it False keeps the user's existing key order intact, which is the lossless-friendly choice). The serialized string is the canonical payload.
      * Route through the ABC helper: `self._write_via_atomic(serialized, mode)`. Do NOT call `atomic_write` directly (D-29). Do NOT call `backup_once` (higher-layer gate).
      * Default `mode=None` (per D-58): D-DEFERRED-01 means `mode=None` yields `0o600` from the tempfile (NOT "preserve existing mode"). Document this in the docstring: `models_cache.json` holds no secret, so `0o600` is more restrictive than the conventional `0o644` but harmless (a more-restrictive mode is never a security regression). Callers MAY pass an explicit `mode` (e.g. `0o644`) if matching the conventional cache-file mode matters — but the default `None` is safe.

    `__all__ = ["JsonBackend", "deep_merge"]`.

    Do NOT add Phase 15 `glm-5.2` entry content, schema, or any model-metadata logic here — this module is the generic merge primitive (D-38 analog: primitives only, no "what does a zai entry look like"). Phase 15 supplies the entry dict.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phase 15 caller → JsonBackend → filesystem (`models_cache.json`) | The user's existing `models_cache.json` entries are precious (other models' metadata); the backend merges, never clobbers. |
| `deep_merge` recursion → untrusted input shape | `content` comes from the caller; a malformed (non-dict) input or a pathological nesting depth could corrupt state or exhaust the stack. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-03 | Tampering (data loss) | JsonBackend.write_canonical (merge-not-overwrite) | high | mitigate | `write_canonical` reads the existing state via `read()` then `deep_merge`s — existing keys are preserved structurally (the merge cannot drop a key the override does not mention). `test_json_write_merges_into_existing` pins that a seeded `glm-4.6` entry survives a `glm-5.2` write. A whole-file overwrite would silently delete user entries; the deep-merge path makes that impossible. |
| T-09-03b | Tampering (corrupt input) | JsonBackend.read / write_canonical type guards | medium | mitigate | `read()` raises `ValueError` if the file parses to a non-object top level; `write_canonical` raises `TypeError` if `content` is not a dict. `deep_merge` raises `TypeError` if either argument is not a dict. These guards fail loudly on corrupt/unexpected input rather than silently coercing. `test_json_read_rejects_non_object_top_level` + `test_json_write_rejects_non_dict_content` pin both. |
| T-09-03c | Denial of Service (recursion depth) | deep_merge recursion | low | accept | `deep_merge` recurses on nested dicts. `models_cache.json` is a flat 2-3 level object (model → metadata); Python's default recursion limit (~1000) is never approached. A maliciously deep file would raise `RecursionError` (loud failure, not corruption). Acceptable for a local CLI operating on the user's own cache file. |
| T-09-03-SC | Tampering | pip install | low | accept | JsonBackend uses only stdlib `json` — no new package introduced (D-61). |
</threat_model>
