    Module-level constants (D-60 — EXACT strings, grep-able sentinels):
    - `MARKER_START = "# >>> zai-codex-helper >>>"`
    - `MARKER_END = "# <<< zai-codex-helper <<<"`
    These are the EXACT strings from D-60; do not alter spacing or punctuation. Export them in `__all__` so Phase 13 uninstall and any grep-based doctor check can reuse the single source of truth.

    Class `ShellBackend(ConfigBackend)`:
    - `__init__(self, paths: Paths) -> None`: `super().__init__(paths, "zshrc")`.
    - `read(self) -> str`: return the WHOLE file text (`self._path.read_text(encoding="utf-8")`) if it exists; if it does not exist, return `""` (an empty .zshrc is the no-block baseline — this is cleaner than raising for a shell file, since a fresh user has no .zshrc yet; the write path handles "file absent" by treating it as empty text). Document this choice (D-57: "reading the whole file + a get_block accessor is cleanest").
    - `get_block(self) -> str | None`: return the text BETWEEN the markers (exclusive of the marker lines) if both markers are present, else None. This is the accessor for callers that want just the helper block.
    - `exists(self) -> bool`: `self._path.exists()`.
    - `write_canonical(self, content: str, mode: int | None = 0o644) -> None`:
        * `content` is the block body (the shell helper text) WITHOUT the markers — the method wraps it in the markers itself. (Default mode 0o644 per D-DEFERRED-01 reasoning above.)
        * Build the fenced block string: a single string joining `MARKER_START`, a newline, the `content`, a newline, `MARKER_END` (each marker on its own line).
        * Read the current file text via `read()` (handles file-absent as "").
        * If the current text already contains BOTH `MARKER_START` and `MARKER_END`: REPLACE the fenced section (from MARKER_START through MARKER_END inclusive) with the new fenced block. Use a regex with `re.DOTALL` matching `re.escape(MARKER_START)` ... `re.escape(MARKER_END)` to locate the existing fence. `re.escape` on the markers is load-bearing — the markers contain `>` and `<` which are regex metacharacters; escaping prevents the fence itself from being treated as a pattern (T-09-02 mitigation).
        * Else (no markers): APPEND the fenced block to the current text (with a leading newline separator if the current text is non-empty and doesn't already end in a newline, so the fence starts on its own line).
        * NEVER duplicate: because the replace branch is a single in-place substitution and the append branch only fires when markers are absent, calling write_canonical twice with the same content yields exactly ONE fence (idempotent — D-57).
        * Route the rewritten whole-file text through the ABC helper: `self._write_via_atomic(rewritten_text, mode)`. The helper writes the ENTIRE .zshrc (block + user content), not just the block — this is what preserves the user's content. Do NOT call atomic_write directly (D-29). Do NOT call backup_once (higher-layer gate).
    - `remove_block(self) -> None`: read the current text; if both markers are present, delete the fenced section (markers + content) leaving the rest of the text intact; rewrite the whole file via `self._write_via_atomic(text_without_fence, 0o644)`. If no markers, this is a no-op (idempotent remove — calling it on a file with no fence leaves the file unchanged). Handle the trailing-newline cleanup so removing the block does not leave a dangling blank line where the fence was (collapse a resulting double-blank-line to single where the fence was removed, to keep the file tidy — but do NOT touch blank lines elsewhere).

    `__all__ = ["ShellBackend", "MARKER_START", "MARKER_END"]`.

    Do NOT add setup/uninstall/doctor logic (Phase 12/13/14). This module is the marker-fence primitive only.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI/domain → ShellBackend → filesystem (.zshrc) | The user's hand-written .zshrc is precious; the backend mutates only the fenced region and must preserve the rest verbatim. |
| Marker-fence regex match → text rewrite | The fence locator uses regex; the markers contain `>`/`<` regex metacharacters that must be escaped to avoid pattern injection or match failure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-02 | Tampering | ShellBackend.write_canonical / remove_block regex | high | mitigate | `re.escape(MARKER_START)` / `re.escape(MARKER_END)` with `re.DOTALL` so the fence locator is a literal match, not a pattern; test_shell_write_twice_is_idempotent + test_shell_preserves_outside_content pin no-duplication and lossless preservation. A malicious or malformed block body cannot escape the fence (the body is inserted verbatim BETWEEN escaped literal markers). |
| T-09-02b | Tampering (user content) | ShellBackend.write_canonical (outside-fence preservation) | high | mitigate | write_canonical rewrites the WHOLE file (user content + fence) via _write_via_atomic; test_shell_preserves_outside_content pins byte-identical survival of seeded user content. |
| T-09-SC | Tampering | pip install | low | accept | ShellBackend uses only stdlib (re, pathlib) — no new package introduced. |
